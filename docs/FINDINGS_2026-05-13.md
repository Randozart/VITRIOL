# VITRIOL Findings — 2026-05-13

> All benchmarks, bottlenecks, failures, and production requirements documented.

---

## 1. Executive Summary

VITRIOL successfully runs Qwen3.6-35B-A3B (35B parameters) on a GTX 1070 Ti (8GB VRAM) at **7.19 tok/s** by using llama.cpp's `-ot ".*exps.*=CPU"` flag to keep all 256 MoE experts on CPU while only the base model (embeddings + 20 attention layers = ~775MB) lives on GPU.

**The bottleneck is confirmed: CPU→GPU expert transfer consumes 60-70% of per-token time.** GPU utilization is only 15-18%.

## 3. Bottleneck Analysis (2026-05-12)

### Per-Token Time Breakdown (35B MoE, estimated)

| Phase | Time | Percentage | Bottleneck? |
|-------|------|------------|-------------|
| Expert loading (CPU→GPU) | ~100-120ms | 60-70% | **YES** |
| Attention compute (GPU) | ~30-40ms | 20-25% | No |
| Expert compute (GPU) | ~20-30ms | 10-15% | No |
| Router + overhead | ~10ms | 5% | No |

### GPU Utilization Analysis
- 35B MoE: 15-18% utilization → **82-85% idle time**
- 9B dense: 25-35% utilization → **65-75% idle time**
- Even the 9B model doesn't saturate the GTX 1070 Ti

### Prompt Processing Gap
- Short prompt (18 tokens): 35B = 7.37 t/s, 9B = 13.92 t/s
- Medium prompt (114 tokens): 35B = 53.65 t/s, 9B = 137.54 t/s
- First-token overhead: 35B = 2.4s, 9B = 1.3s
- **LLAMA EXPLANATION**: 35B has 12GB of model on CPU; the first token requires loading ALL 256 experts into CPU RAM, plus warming up the GPU compute graph. The 9B model fits entirely in GPU VRAM (5.3GB) and requires no CPU staging.

### KV Cache Degradation (35B MoE)
- 50 tokens: 6.16 tok/s
- 200 tokens: 5.53 tok/s
- KV cache is on CPU (2560 MB), adding latency as context grows

## 4. The Critical Code Path: Expert Loading During Inference

Investigation of llama.cpp source code (2026-05-13) reveals the exact path:

### Phase 1: Model Loading (one-time)

```
load_tensors() [llama-model-loader.cpp]
  -> ml.create_tensor() [llama-model-loader.cpp:1045]
    -> buft_for_tensor() [llama-model-loader.cpp:1078]
      -> checks tensor_buft_overrides (-ot flag regex match) [line 1156]
      -> select_weight_buft() [line 1184]
    -> ggml_backend_alloc_ctx_tensors()
      -> ggml_backend_cuda_buffer_set_tensor() [ggml-cuda.cu:678]
        -> cudaMemcpyAsync(host→device, cudaStreamPerThread) [LINE 682]
        -> cudaStreamSynchronize()
```

### Phase 2: Inference (per-token, the critical path)

```
build_moe_ffn() [llama-graph.cpp:1310]
  -> build_lora_mm_id(up_exps/gate_exps/down_exps) [line 1005]
    -> ggml_mul_mat_id(ctx0, w, cur, ids) [line 1009]

ggml_backend_sched_compute_splits() [ggml-backend.cpp:1541]
  -> For MUL_MAT_ID ops with host-staged weights [lines 1576-1660]:
    -> Reads ids tensor from GPU (copies selected expert indices to host) [line 1606]
    -> Determines which experts are used [lines 1610-1618]
    -> For each group of consecutive used experts [lines 1623-1660]:
      -> ggml_backend_tensor_set_async() [line 1630]
        -> ggml_backend_cuda_set_tensor_async() [ggml-cuda.cu:2986]
          -> cudaMemcpyAsync(host→device, cuda_ctx->stream()) [LINE 2992]

ggml_cuda_graph_compute() [ggml-cuda.cu:4288]
  -> ggml_cuda_compute_forward() [line 2629]
    -> case GGML_OP_MUL_MAT_ID: [line 2821]
      -> ggml_cuda_mul_mat_id() [line 2471]
        -> For each expert [lines 2576-2621]:
          -> ggml_cuda_mul_mat(src0_slice, src1_slice, dst_slice) [line 2616]
```

### The Two Key cudaMemcpyAsync Locations

| Location | File:Line | When Called | Stream | Sync? |
|----------|-----------|-------------|--------|-------|
| `ggml_backend_cuda_buffer_set_tensor` | ggml-cuda.cu:682 | Model loading (one-time) | `cudaStreamPerThread` | Synchronized |
| `ggml_backend_cuda_set_tensor_async` | ggml-cuda.cu:2992 | **Every token (expert slices)** | `cuda_ctx->stream()` | Async |

### The Scheduler's MoE Optimization (ggml-backend.cpp:1576-1660)

This is the **most important code** for VITRIOL. When the scheduler detects a `GGML_OP_MUL_MAT_ID` node with a CPU-hosted weight tensor, it:
1. Reads the `ids` tensor from GPU (which experts are selected for this token)
2. Reorders the selected expert indices
3. Calculates slices (byte offsets + sizes) for each group of consecutive experts
4. Calls `ggml_backend_tensor_set_async` to copy only the **expert slices** from CPU to GPU

**This already implements sparse expert selection at the data movement level** — only the 8 active experts are transferred per token, not all 256.

## 5. VITRIOL Hook Integration Status (2026-05-13)

### Files
| File | Status |
|------|--------|
| `/mnt/data/ai/llama.cpp/include/vitriol-config.h` | Complete (modes, config struct) |
| `/mnt/data/ai/llama.cpp/src/vitriol-config.cpp` | Complete (env var parsing) |
| `/mnt/data/ai/llama.cpp/ggml/src/ggml-cuda/vitriol-cuda-integration.h` | Complete (hook declarations) |
| `/mnt/data/ai/llama.cpp/ggml/src/ggml-cuda/vitriol-cuda-integration.cpp` | **ALL STUBS** (returns false, never called) |

### Critical Finding
`vitriol_cuda_set_tensor_hook` is **NEVER CALLED** from any code path. A grep for `vitriol` in `ggml-cuda.cu` returns zero matches. The hooks exist as stubs but are not wired into:
- `ggml_backend_cuda_buffer_set_tensor` (line 678)
- `ggml_backend_cuda_set_tensor_async` (line 2986)
- `ggml_backend_cuda_buffer_set_tensor_2d` (line 694)

### What Would Need to Change
1. **Wire the hook**: Call `vitriol_cuda_set_tensor_hook()` before `cudaMemcpyAsync` at lines 678 and 2992. If hook returns true (VITRIOL handled transfer), skip standard copy.
2. **Implement DMA**: In `vitriol_cuda_set_tensor_hook`, when mode is `VITRIOL_MODE_STREAM`, perform NVMe→GPU DMA via kernel module or GPUDirect Storage.
3. **Double-buffer**: Use `buffer_a`/`buffer_b` (already declared) with separate CUDA streams for prefetching next token's experts.
4. **Expert-aware**: The hook receives the tensor name and file offset. Use this to DMA only the expert slices needed (read `ids` tensor from GPU to determine which experts).

## 6. Speculative Decoding Failure Analysis (2026-05-13)

### Attempt 1: Same-GPU Draft (Qwen3-0.6B on GTX 1070 Ti)

| Metric | Value |
|--------|-------|
| Draft model | Qwen3-0.6B-Q4_K_M (379 MB) |
| Target model | Qwen3.6-35B-A3B |
| Total VRAM | 5449 MB |
| Effective tok/s | **3.70** (worse than 7.19 baseline) |

**Causes:**
1. **Tokenizer mismatch**: Qwen3-0.6B vocab = 151,936 tokens; Qwen3.6-35B-A3B vocab = 248,320 tokens. llama.cpp reports: "the target and draft vocabs are not compatible — tokens will be translated." Translation is lossy; acceptance rate near zero.
2. **GPU contention**: Both models on the same GPU. Draft model uses compute resources that compete with the target. At 15% target GPU utilization, there's room, but the draft is still overhead.
3. **No compatible small Qwen3.6 model exists**: Qwen3.6 family currently starts at 35B. No 0.5B version available.

### Attempt 2: GTX 960 Draft (not tested due to binary mismatch)

**Blockers:**
1. GTX 960 is sm_52 (Maxwell). llama.cpp was compiled for sm_61 (Pascal). CUDA error: "no kernel image is available for execution on the device."
2. **Fix**: Rebuild llama.cpp with `-DGGML_CUDA_MMV_ARCH="sm_52;sm_61"` (requires CUDA toolkit supporting sm_52).
3. **Verdict**: Not worth pursuing. The 960 is a hardware testbench, not a daily driver. Rebuild effort isn't justified.

### Recommendation: N-gram Speculation (no download, no rebuild)

llama.cpp supports `--speculative-ngram` for lookahead decoding without a draft model. Reuses previously generated tokens as draft candidates. Expected speedup: 10-30% for zero cost.

## 7. Alka Commands Required for Production DMA Path (2026-05-13)

### CRITICAL — Must be production-ready before VITRIOL can use Alka for DMA

| Command | OpCode | Purpose | Production Needed? | Status |
|---------|--------|---------|--------------------|--------|
| **FLOW** | 0x03 | NVMe→GPU expert transfer via DMA | **CRITICAL** — replaces `cudaMemcpyAsync` | Tool exists, needs real DMA impl |
| **CLAIM** | 0x01 | Claim GPU/NVMe hardware, unbind driver | **CRITICAL** — required before any DMA | Tool exists, validated |
| **FENCE** | 0x05 | Wait for DMA completion, poll metapage | **CRITICAL** — ensures data arrived before compute | Tool exists, mock only |
| **SHIFT** | 0x04 | BAR1 sliding window remap (256MB aperture) | **CRITICAL** — required for >256MB transfers | Tool exists, mock only |
| **SIGNAL** | 0x09 | Trigger GPU compute after experts loaded | **CRITICAL** — coordinates load→compute pipeline | Tool exists, validated |
| **LIMIT** | 0x0E | Thermal guard (halt at 85C) | **REQUIRED** — safety for sustained DMA | Tool exists, validated |
| **SENSE** | 0x07 | Monitor GPU temperature during streaming | **REQUIRED** — thermal awareness | Tool exists, mock only |

### IMPORTANT — Needed for full pipeline, but can be deferred

| Command | OpCode | Purpose | Priority | Status |
|---------|--------|---------|----------|--------|
| **REFRACT** | 0x3B | Sub-tensor slicing for 256MB BAR window | High — needed for >256MB transfers | Tool exists, validated |
| **SYNC** | 0x06 | Memory barrier (L1/L2/L3) | High — memory ordering | Tool exists, validated |
| **PIPE** | 0x3C | Continuous DMA ring buffer for prefetch | Medium — performance optimization | Tool exists, mock only |
| **REVERT** | 0x0D | Rollback state on failure | Medium — safety/recovery | Tool exists, mock only |
| **SNAP** | 0x0C | Snapshot GPU state before DMA | Medium — checkpoint for recovery | Tool exists, mock only |
| **WATCH** | 0x2F | Monitor throughput during streaming | Low — observability | Tool exists |
| **TRACE** | 0x30 | Profile execution timing | Low — debugging | Tool exists |
| **GUARD** | 0x31 | Runtime safety sentinel | Low — redundant with LIMIT | Tool exists |
| **VOID** | 0x1F | Secure erase transferred data (Azoth rollback) | Low — cleanup | Tool exists, mock only |

### NOT NEEDED for initial DMA path

| Command | Reason |
|---------|--------|
| **STAKE**, **MOLT**, **FOSSILIZE** | Memory/allocation not needed for streaming approach |
| **STRIKE**, **VEIL**, **GHOST**, **HIJACK** | Security subversion, not needed for inference |
| **OSSIFY**, **BOND**, **IMC_HIJACK** | Substrate orchestration, future optimization |
| **FORGE**, **RECAST**, **CRYSTALLIZE** | FPGA operations, not relevant to GPU DMA |
| **QUENCH**, **OVERCLOCK** | Power management, beyond current scope |
| **SPECULATE** | Not implemented in Alka compiler |
| **DRIFT**, **RESONATE**, **OSCILLATE**, **STILL** | Hardware timing, future optimization |

### Minimum Viable Instruction Set for DMA Path

```
CLAIM GPU_1070TI;          // Bind GPU, get BAR0/BAR1 access
CLAIM NVME_SSD;            // Get NVMe block device access
LIMIT GPU_1070TI 85000;    // Thermal safety

// For each expert transfer:
SHIFT GPU_1070TI.DATA_PLANE @ offset;  // Window to correct BAR1 offset
FLOW NVME_SSD[expert_addr] -> GPU_1070TI.VRAM[gpu_addr] expert_size;
FENCE GPU_1070TI.METAPAGE == token;     // Wait for DMA completion

// Barrier + signal:
SYNC L3;
SIGNAL COMPUTE_START;

// Safety:
SENSE GPU_1070TI.THERMAL AS temp;
```

**Currently working in Alka compiler:** CLAIM, FLOW, SHIFT, FENCE, SYNC, SIGNAL, LIMIT, SENSE, REFRACT, PIPE (compilation + mock execution).

**Needs real hardware implementation:** FLOW (NVMe→GPU DMA), SHIFT (BAR1 remap), FENCE (GPU metapage polling), PIPE (ring buffer).

## 8. Step 2 Analysis: Double-Buffer / Prefetch (2026-05-13)

### Finding: Not worth implementing yet

The analysis of `ggml-backend.cpp:1576-1660` reveals that llama.cpp's scheduler **already implements a form of sparse expert selection** — it reads the `ids` tensor to determine which experts are active, then copies only those expert slices from CPU to GPU.

However, this happens **synchronously in the graph evaluation path**. The GPU waits for the copy before computing.

### Why Prefetch Won't Help Yet
- GPU utilization is 15-18% — there's nothing to overlap with because the GPU is already mostly idle
- Prefetch hides latency, but the problem is bandwidth, not latency
- Until the CPU→GPU transfer is optimized (via DMA), double-buffering just means two slow transfers instead of one

### When Prefetch Will Matter
After DMA is implemented and GPU utilization increases to 50%+, prefetch becomes the next optimization.
